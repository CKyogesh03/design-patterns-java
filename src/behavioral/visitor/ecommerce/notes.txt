A good way to understand the Visitor Pattern is through a real production-style scenario where:

You already have a fixed set of object types
You need to add multiple operations on them
You don’t want to keep modifying those classes every time
Real-Time Scenario: E-Commerce Order Items

Imagine you are building an e-commerce system.

You have different item types:

Book
Electronics
Grocery

Now business keeps adding operations like:

Tax calculation
Discount calculation
Invoice generation
Export to Excel
Analytics reporting

If you put all logic inside item classes, classes become huge and violate SRP.

Visitor pattern solves this.

Without Visitor Pattern
class Book {
    double getTax() {}
    double getDiscount() {}
    void exportToExcel() {}
    void generateInvoice() {}
}

Problem:

Every new operation changes all classes
Hard to maintain
Business logic scattered

=============================================================

Why Companies Use Visitor Pattern

Very useful when:

Scenario	Example
Report generation	-> PDF, Excel, CSV export
Compilers	-> AST node processing
Payment systems	-> Different payment processing logic
Insurance systems	-> Premium calculators
E-commerce	-> Tax, discount, invoice
SAP/Hybris	-> Product/cart calculation engines

Actual Production Example (Compiler)

Java compiler internally uses Visitor heavily.

Example:

IfStatement node
ForLoop node
Variable node

Visitors:

Syntax validation visitor
Optimization visitor
Bytecode generation visitor

Compiler adds new operations without modifying AST classes.

Key Interview Point:
Visitor pattern is best when:
Object structure is stable

BUT:
Operations change frequently
Major Advantage:
You can add new business operations without touching existing classes.

Add:
GSTVisitor
AuditVisitor
FraudCheckVisitor

No change in:
Book
Electronics

This follows:
Open Closed Principle
Single Responsibility Principle
Important Limitation

Adding NEW element types is difficult.
If you add:
Furniture

Then ALL visitors must change:
visit(Furniture furniture)

So Visitor is good when:
Element classes are stable
Operations are growing frequently
Easy Memory Trick
Visitor Pattern

“Same objects, different operations.”

Objects stay same.
Visitors keep changing.

eg: order/cart model object is send to different visitors for different operations
